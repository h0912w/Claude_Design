# 단종 회로소자 대체품 적합성 판단 자동화 시스템 통합 설계서

## 0. 문서 목적
본 문서는 **단종 또는 수급 불안정 회로소자에 대해, 사용자가 미리 선정한 대체품 후보가 실제로 적합한지 판단하는 자료를 자동 생성하는 Claude Code 기반 시스템**의 통합 설계서이다.  
최종 구현 시 Claude Code 프로젝트의 기준 문서로 사용하며, 구현 상세 코드는 포함하지 않고 **구조, 역할, 워크플로우, 검증 및 산출물 정의** 수준까지만 다룬다.

---

## 1. 작업 컨텍스트 문서

### 1.1 배경
회로 설계 및 부품 선정 과정에서는 다음과 같은 문제가 자주 발생한다.

- 기존 설계에 사용된 부품이 단종(EOL)되거나 공급 불안정 상태가 됨
- 설계자가 대체품 후보를 직접 선정해도, 실제로 완전 호환인지 판단하기 어려움
- 데이터시트 내 스펙이 많고 분산되어 있어, 사람 검토에 많은 시간이 소요됨
- 단순 수치 비교뿐 아니라, 조건부 제약, 주석, 그래프, 패키지, 핀 구성, 권장 동작 영역, 절대 최대 정격, 기능 차이까지 함께 봐야 함

따라서 본 시스템은 **기존 부품과 대체품 후보의 데이터시트를 면밀히 분석하여, 모든 스펙을 빠짐없이 추출하고, 그 근거 원문/원도표를 원본 그대로 CROP하여, 사람이 바로 검토 가능한 비교 자료를 자동 생성**하는 것을 목표로 한다.

### 1.2 목적
본 시스템의 목적은 다음과 같다.

1. 기존 부품 데이터시트에서 **모든 스펙을 100% 빠짐없이 추출**한다.
2. 각 스펙마다 **근거가 되는 데이터시트 원문/표/그래프/도면 일부를 원본 그대로 CROP**하여 연결한다.
3. 대체품 후보 데이터시트에 대해서도 동일한 작업을 수행한다.
4. 두 부품의 스펙을 **일대일 비교표**로 정리하고, 동일/차이/부족/주의 항목을 식별한다.
5. AI가 각 항목별로 대체 적합성을 판단하고, 최종적으로 **종합 의견서(Word)**를 작성한다.
6. 필요 시 웹검색을 활용해 **제조사 권장 대체품 정보, PCN/EOL/NRND 여부, 공급상태 관련 참고정보**를 첨부한다.

### 1.3 범위
#### 포함 범위
- 디지털 PDF 형태의 데이터시트 입력
- 기존 부품 1개 + 대체품 후보 1개를 기본 단위로 처리
- Part number는 데이터시트 파일명에서 식별
- 데이터시트 전 영역에서 스펙/조건/주석/도표/패키지/핀정보/전기특성/타이밍/열특성 등 추출
- 스펙 근거 이미지 CROP
- Excel 3개 시트 생성
- Word 종합 판단 보고서 생성
- 제조사/공식 채널 중심 웹검색 기반 보강 정보 수집

#### 제외 범위
- 회로 전체 맥락(주변 회로, 실제 보드 동작점, 펌웨어 제약)까지 자동 판단
- SPICE 시뮬레이션 자동 수행
- 실측 검증 자동화
- 스캔본 OCR 중심 처리 최적화
- 여러 대체품 후보를 한 번에 순위화하는 다중 후보 비교 기능의 상세 구현

> 기본 설계는 1:1 비교 기준으로 작성하되, 이후 N:1 비교 구조로 확장 가능하도록 설계한다.

### 1.4 입력 정의
시스템 입력은 지정된 경로 구조를 따른다.

```text
/project-root
 ├── /input
 │   ├── /original
 │   │   └── <기존부품_정확한PartNumber>.pdf
 │   └── /candidate
 │       └── <대체부품_정확한PartNumber>.pdf
```

#### 입력 규칙
- 파일명은 반드시 실제 구매에 사용하는 **정확한 Part number 풀 네임**이어야 한다.
- 파일 형식은 원칙적으로 텍스트 추출 가능한 디지털 PDF를 우선 지원한다.
- PDF 내부 언어는 영어를 기본 가정하되, 제조사 데이터시트 특성상 다국어 일부 포함 가능.
- 데이터시트 페이지 수, 표 구조, 레이아웃, 그래프 유형은 제조사마다 다를 수 있다.

### 1.5 출력 정의
시스템 출력은 다음과 같다.

```text
/project-root
 ├── /output
 │   ├── comparison_report.xlsx
 │   ├── substitution_assessment.docx
 │   ├── /crops
 │   │   ├── original
 │   │   └── candidate
 │   ├── /qa
 │   │   ├── /input
 │   │   │   ├── /original
 │   │   │   └── /candidate
 │   │   ├── /artifacts
 │   │   └── /reports
 │   ├── /intermediate
 │   │   ├── original_specs.json
 │   │   ├── candidate_specs.json
 │   │   ├── comparison_matrix.json
 │   │   └── evidence_index.json
 │   └── /logs
 │       └── run_log.md
```

#### Excel 파일
- **1번 시트:** 비교 요약 시트
- **2번 시트:** 기존 부품 스펙 추출 시트
- **3번 시트:** 대체품 후보 스펙 추출 시트

#### Word 파일
- 종합 판단 보고서
- AI가 각 차이 항목의 의미, 사용 가능 여부, 주의사항, 추가 검토 필요사항을 설명
- 제조사 권장 대체품/공식 공지/수급 참고 정보 첨부

### 1.6 주요 제약조건
1. **모든 스펙을 100% 빠짐없이 추출**하는 것을 목표로 한다.
2. 사람이 검토하기 쉬워야 하며, 이를 위해 **원본 근거 CROP**를 각 항목에 연결해야 한다.
3. CROP 이미지는 **절대 화질이 깨지지 않아야 하며**, 가로세로 비율 보정보다 **원본 보존**을 우선한다.
4. 스크립트 사용은 최소화하되, **결정론적 작업(파일 I/O, PDF crop, 엑셀 삽입, 문서 생성)**은 스크립트로 처리한다.
5. 판단/분류/비교/리스크 해석/종합 의견 작성은 Claude Code 에이전트가 중심이 된다.
6. 서브에이전트 구조를 채택한다.
7. 최종 출력은 사람이 근거를 추적할 수 있어야 하며, 각 판단은 가능한 한 근거 페이지/크롭 참조를 가져야 한다.
8. 사용자가 명시적으로 요청했거나, 코드 또는 문서(CLAUDE.md, AGENT.md, SKILL.md, 설계 문서, 보고서 템플릿 등)가 수정된 경우에는 **반드시 QA 전체 실행**을 수행해야 한다.
9. QA는 별도 테스트 전용 코드가 아니라, 실제 사용자 산출물을 만드는 동일 실행 경로와 동일 코드를 사용해야 한다.

### 1.7 성공 기준
본 시스템은 아래 조건을 만족할 때 성공으로 본다.

1. 기존 부품과 대체품 데이터시트에서 **누락 없이 스펙 항목이 구조화**되어 있다.
2. 각 스펙 항목에는 **최소 1개 이상의 근거 crop 또는 근거 텍스트 위치 정보**가 연결되어 있다.
3. 비교 시트에서 **동일/차이/불확실/미기재**가 명확히 표기된다.
4. AI 판단이 **왜 그런 결론인지 사람이 즉시 검토 가능**한 수준으로 설명된다.
5. Word 보고서가 단순 찬반이 아니라, **대체 가능 / 조건부 가능 / 비추천 / 추가 검증 필요**로 구분하여 실무적 판단을 제공한다.
6. QA 실행 시 샘플 데이터시트를 사용해 실제 전체 시스템을 끝까지 실행하고, 생성된 Excel/Word를 다시 읽어 Plan 문서와 대조했을 때 누락 요소가 없어야 한다.
7. QA는 실제 데이터시트와 대조하여 **정말 모든 스펙이 추출되었는지**를 검증해야 한다.
8. QA는 근거 이미지가 단순히 셀 옆에 배치된 것이 아니라, **엑셀의 해당 셀 내부에 삽입/고정되어 함께 정렬되었는지**를 검증해야 한다.
9. QA 실패 시 문제 항목, 원인 추정, 수정 권고가 구조화된 피드백으로 남아 다음 코딩 루프의 입력으로 사용된다.

### 1.8 용어 정의
- **기존 부품(original part):** 현재 설계에 사용 중인 기준 부품
- **대체품(candidate part):** 사용자가 후보로 제시한 대체 부품
- **스펙(spec):** 데이터시트에 정의된 수치, 조건, 기능 설명, 그래프 기반 특성, 표 주석, 패키지/핀/절대정격/권장조건 등 비교 가능한 모든 속성
- **근거(evidence):** 특정 스펙을 추출한 근거가 되는 데이터시트의 문장, 표, 그림, 그래프, 도면 일부
- **CROP:** 원본 PDF에서 근거 영역을 해상도 저하 없이 잘라낸 이미지
- **비교 매트릭스(comparison matrix):** original과 candidate의 각 스펙 항목을 1:1 정렬한 구조화 데이터
- **대체 적합성 판단:** candidate가 original을 실제 설계에 대체 가능한지에 대한 항목별/종합적 평가

## 2. 워크플로우 정의

## 2.1 전체 흐름

```text
[입력 검증]
   ↓
[기존 부품 데이터시트 분석]
   ↓
[기존 부품 스펙 전체 추출]
   ↓
[기존 부품 근거 crop 생성]
   ↓
[대체품 데이터시트 분석]
   ↓
[대체품 스펙 전체 추출]
   ↓
[대체품 근거 crop 생성]
   ↓
[스펙 정규화 및 1:1 매핑]
   ↓
[항목별 비교 및 적합성 판단]
   ↓
[공식/제조사 웹 보강 정보 수집]
   ↓
[Excel 생성]
   ↓
[Word 종합 보고서 생성]
   ↓
[산출물 검증 및 로그 기록]
   ↓
[QA 트리거 판정]
   ↓
[필수 QA 실행(조건 충족 시)]
   ↓
[QA 통과 또는 수정 피드백 생성]
```

## 2.2 상태 전이
- `INIT` → 입력 파일 확인 완료 시 `INPUT_VALIDATED`
- `INPUT_VALIDATED` → 기존 부품 파싱 성공 시 `ORIGINAL_PARSED`
- `ORIGINAL_PARSED` → 기존 부품 crop 생성 완료 시 `ORIGINAL_EVIDENCED`
- `ORIGINAL_EVIDENCED` → 대체품 파싱 성공 시 `CANDIDATE_PARSED`
- `CANDIDATE_PARSED` → 대체품 crop 생성 완료 시 `CANDIDATE_EVIDENCED`
- `CANDIDATE_EVIDENCED` → 비교 매트릭스 생성 완료 시 `COMPARED`
- `COMPARED` → 외부 보강 정보 수집 완료 시 `ENRICHED`
- `ENRICHED` → Excel/Word 생성 완료 시 `DOCUMENTED`
- `DOCUMENTED` → 산출물 검증 완료 시 `VALIDATED`
- `VALIDATED` → QA 트리거 없음 시 `DONE`
- `VALIDATED` → QA 필요 시 `QA_PENDING`
- `QA_PENDING` → QA 실행 완료 시 `QA_DONE`
- `QA_DONE` → QA 통과 시 `DONE`
- `QA_DONE` → 보완 필요 시 `REWORK_REQUIRED`
- `REWORK_REQUIRED` → 수정 완료 후 `QA_PENDING`
- 어느 단계에서든 치명적 실패 발생 시 `FAILED`
- 선택적 보강 정보 수집 실패 시 `PARTIAL_DONE`

## 2.3 단계별 상세 정의

### 단계 1. 입력 검증
**목적:** 입력 폴더 구조와 파일명을 점검하고, original/candidate PDF를 식별한다.

**입력:** `/input/original/*.pdf`, `/input/candidate/*.pdf`

**출력:** `input_manifest.json`

**에이전트 판단 vs 코드 처리**

| 업무 | 사용할 에이전트 | 스크립트로 처리 |
|---|---|---|
| 파일명이 정확한 Part number 형식인지 해석 | main-orchestrator | 디렉터리 탐색, 파일 존재 확인, 확장자 확인 |
| original/candidate 역할 구분 확인 | main-orchestrator | manifest 파일 생성 |

**성공 기준:** original 1개, candidate 1개가 식별되고 manifest가 생성된다.

**검증 방법:** 규칙 기반 검증

**실패 시 처리:** 자동 재시도 1회 후, 여전히 실패하면 에스컬레이션

---

### 단계 2. 기존 부품 데이터시트 구조 분석
**목적:** 기존 부품 데이터시트의 전체 페이지를 훑어 섹션 구조를 파악하고, 스펙 후보가 존재하는 영역을 최대한 누락 없이 식별한다.

**입력:** original PDF

**출력:** `original_section_map.json`

**에이전트 판단 vs 코드 처리**

| 업무 | 사용할 에이전트 | 스크립트로 처리 |
|---|---|---|
| 데이터시트 섹션 분류 (features, absolute max, recommended conditions, electrical characteristics, package, pin config, timing, thermal, notes 등) | spec-extractor | PDF 텍스트 블록/좌표 추출, 페이지 이미지 렌더링 |
| 표/그래프/도면/주석이 스펙 추출 대상인지 판단 | spec-extractor | 페이지 인덱싱 |

**성공 기준:** 데이터시트 전 페이지에 대해 스펙 가능 섹션이 누락 없이 맵핑된다.

**검증 방법:** LLM 자기 검증 + 규칙 기반 검증(모든 페이지가 section map에 포함되어야 함)

**실패 시 처리:** 자동 재시도 2회, 여전히 불확실하면 에스컬레이션

---

### 단계 3. 기존 부품 스펙 전체 추출
**목적:** 기존 부품 데이터시트에서 모든 스펙을 구조화된 항목으로 정리한다.

**입력:** original PDF, `original_section_map.json`

**출력:** `original_specs.json`

**에이전트 판단 vs 코드 처리**

| 업무 | 사용할 에이전트 | 스크립트로 처리 |
|---|---|---|
| 표/문장/주석/그래프로부터 스펙 후보를 식별 | spec-extractor | 표 텍스트 추출, 좌표 연결 |
| 서로 다른 표현을 하나의 스펙 키로 통합할지 판단 | spec-normalizer | 원시 추출 결과 저장 |
| 조건부 스펙, footnote, test conditions의 중요도 판단 | spec-extractor | 추출 위치(page, bbox) 기록 |
| 패키지/핀정보/절대정격/권장동작조건/전기특성/타이밍/열특성 등 카테고리 분류 | spec-normalizer | JSON 구조 생성 |

**성공 기준:** 데이터시트의 모든 비교 대상 스펙이 누락 없이 JSON으로 정리된다.

**검증 방법:**
- 규칙 기반: 필수 카테고리 존재 여부 확인
- LLM 자기 검증: 페이지별 재검토를 통해 누락 의심 항목 탐지
- 사람 검토: 필요 시 샘플링 검토

**실패 시 처리:** 자동 재시도 최대 2회 후 에스컬레이션

### 단계 4. 기존 부품 근거 crop 생성
**목적:** 추출된 모든 스펙에 대해 근거가 되는 원본 영역을 고해상도로 CROP한다.

**입력:** original PDF, `original_specs.json`

**출력:** `/output/crops/original/*`, `original_evidence_index.json`

**에이전트 판단 vs 코드 처리**

| 업무 | 사용할 에이전트 | 스크립트로 처리 |
|---|---|---|
| 근거 영역이 단일 문장/표 일부/그래프/도면 중 무엇인지 판단 | evidence-cropper | bbox 기준 PDF crop 렌더링 |
| 여러 요소를 함께 crop해야 의미가 보존되는지 판단 | evidence-cropper | 고해상도 이미지 생성, 파일 저장 |
| 원본 그대로 보존할지 주변 여백을 포함할지 판단 | evidence-cropper | crop 파일명 규칙 적용, index 생성 |

**성공 기준:** 모든 스펙 항목에 대해 최소 1개의 crop이 연결된다.

**검증 방법:**
- 규칙 기반: spec count와 evidence count 매핑 확인
- LLM 자기 검증: crop이 해당 스펙의 의미를 충분히 보여주는지 확인

**실패 시 처리:** 자동 재시도 2회, 그래도 실패하면 스킵+로그 금지. 반드시 에스컬레이션

> 본 단계는 본 시스템의 핵심 단계이므로 선택적 스킵을 허용하지 않는다.

### 단계 8. 스펙 정규화 및 1:1 매핑
**목적:** original과 candidate의 스펙 항목명을 정규화하고, 비교 가능한 단위로 1:1 정렬한다.

**입력:** `original_specs.json`, `candidate_specs.json`

**출력:** `comparison_matrix.json`

**에이전트 판단 vs 코드 처리**

| 업무 | 사용할 에이전트 | 스크립트로 처리 |
|---|---|---|
| 서로 다른 데이터시트 표현을 동등 스펙으로 매핑 | spec-normalizer | 후보 매핑 테이블 저장 |
| 비교 불가 항목/미기재 항목/복수 매핑 충돌 해결 | comparison-judge | 정렬된 matrix 직렬화 |
| 단위 차이(V/mV, A/mA, ns/us 등) 해석 | spec-normalizer | 수치 단위 환산 |

**성공 기준:** 비교 대상 스펙이 누락 없이 1:1 또는 명시적 unmatched 상태로 정리된다.

**검증 방법:**
- 규칙 기반: 모든 original spec이 matrix에 존재해야 함
- LLM 자기 검증: 의미상 잘못 짝지어진 항목이 없는지 검토

**실패 시 처리:** 자동 재시도 2회 후 에스컬레이션

### 단계 9. 항목별 비교 및 적합성 판단
**목적:** 각 스펙 항목에 대해 동일/차이/우위/열위/불확실을 판정하고, 대체 적합성을 판단한다.

**입력:** `comparison_matrix.json`

**출력:** `comparison_assessment.json`

**에이전트 판단 vs 코드 처리**

| 업무 | 사용할 에이전트 | 스크립트로 처리 |
|---|---|---|
| 각 항목이 동일/다름/부족/우위인지 판단 | comparison-judge | 수치 차이 계산, threshold 비교 |
| 차이가 실제 대체 가능성에 영향을 주는지 해석 | equivalence-judge | 결과 저장 |
| 조건부 허용 여부(예: 절대정격은 높지만 권장조건 다름) 판단 | equivalence-judge | 정량 비교 보조 |
| 핀 호환/패키지 호환/기능 차이의 리스크 해석 | equivalence-judge | 기계적 문자열 비교 보조 |

**성공 기준:** 모든 매핑된 항목에 대해 판정 결과와 이유가 존재한다.

**검증 방법:** LLM 자기 검증 + 사람 검토(고위험 항목)

**실패 시 처리:** 자동 재시도 1회 후 에스컬레이션

### 단계 14. 필수 QA 실행 및 재코딩 피드백
**목적:** QA 트리거 조건을 만족하는 경우, 샘플 데이터시트를 이용해 실제 사용자용 코드 경로 그대로 전체 시스템을 실행하고, 생성 산출물을 다시 읽어 누락/오류를 찾고 수정 피드백을 만든다.

**입력:**  
- `/output/qa/input/original/*.pdf`  
- `/output/qa/input/candidate/*.pdf`  
- 최신 Plan 문서 및 설계 문서  
- 실제 사용자 산출물 생성 코드 전체

**출력:**  
- `/output/qa/artifacts/comparison_report.xlsx`  
- `/output/qa/artifacts/substitution_assessment.docx`  
- `/output/qa/reports/qa_validation_report.md`  
- `/output/qa/reports/rework_feedback.json`

**에이전트 판단 vs 코드 처리**

| 업무 | 사용할 에이전트 | 스크립트로 처리 |
|---|---|---|
| QA 대상 실행이 필요한지 판정 | main-orchestrator | 변경 감지, QA 실행 플래그 계산 |
| 샘플 데이터시트가 Plan 검증에 충분한지 판단 | qa-reviewer | QA 입력 폴더 존재/파일 확인 |
| 실제 전체 시스템을 동일 코드로 실행 | qa-reviewer | 사용자용 실행 명령 그대로 수행 |
| 생성된 Excel/Word를 다시 읽고 누락/오류/불일치 판단 | qa-reviewer | 산출물 파일 읽기, 표/이미지 개수 수집 |
| Plan 문서 기준 누락 요소 탐지 | qa-reviewer | 비교용 체크리스트 저장 |
| 수정이 필요한 경우 재코딩 피드백 작성 | qa-reviewer | 피드백 JSON/MD 저장 |
| 피드백을 다음 수정 루프로 전달 | main-orchestrator | 재실행 큐 또는 수정 지시 파일 생성 |

**성공 기준:**
1. QA가 실제 사용자용 실행 경로와 동일한 코드로 수행된다.
2. 샘플 입력으로 Excel/Word 산출물이 정상 생성된다.
3. QA 에이전트가 생성된 산출물을 다시 읽어 Plan 문서 기준 누락 요소가 없음을 확인한다.

**검증 방법:**
- 규칙 기반: QA 입력 존재, 산출물 생성 여부, 시트/문서 구조 확인

- 사람 검토: 최종 릴리스 전 선택 적용 가능

**실패 시 처리:**
- 산출물 미생성/형식 오류: 자동 재시도 1회
- 누락/판단 오류 발견: 스킵 금지, `REWORK_REQUIRED` 상태로 전이 후 재코딩 피드백 생성
- QA 입력 부적절: 에스컬레이션

---
## 2.5 핵심 분기 조건
1. **텍스트 추출이 충분한가?**  
   - 예: 텍스트 기반 파싱 진행  
   - 아니오: 페이지 렌더링 + 시각 기반 분석 비중 증가

2. **해당 스펙이 표/문장만으로 충분한가?**  
   - 예: 단일 crop  
   - 아니오: 표 + footnote 또는 그래프 + 축 범위를 함께 crop

3. **동등 매핑이 명확한가?**  
   - 예: 자동 비교  
   - 아니오: `불확실 매핑`으로 표시하고 사람 검토 플래그 부여

4. **차이가 실제 대체 리스크를 만드는가?**  
   - 예: 위험 또는 비추천 판정  
   - 아니오: 조건부 허용 또는 문제 없음 판정

5. **공식 보강 정보가 존재하는가?**  
   - 예: 보고서에 첨부  
   - 아니오: 공식 정보 미발견으로 기록

## 3.3 에이전트 구조
### 메인 에이전트
- **이름:** `main-orchestrator`
- **역할:** 전체 흐름 조율, 단계별 서브에이전트 호출, 실패 처리, 최종 승인
- **특징:** 서브에이전트 간 직접 호출 금지. 모든 호출은 main-orchestrator를 통해 조율

### 서브에이전트 구성

#### 1) spec-extractor
- **역할:** 데이터시트 전체 분석, 스펙 후보 추출, 섹션 파악
- **입력:** PDF, section map 관련 중간 데이터
- **출력:** 원시 스펙 목록, 섹션 맵
- **데이터 전달 방식:** 파일 기반 (`/output/intermediate/*.json`)

#### 2) spec-normalizer
- **역할:** 스펙명 정규화, 단위 통합, 비교 가능 구조 정리
- **입력:** 원시 스펙 JSON
- **출력:** 정규화된 스펙 JSON, 매핑 후보
- **데이터 전달 방식:** 파일 기반

#### 3) evidence-cropper
- **역할:** 스펙 근거 영역 판정 및 crop 전략 결정
- **입력:** 스펙 JSON, PDF 좌표 정보
- **출력:** crop index, crop image path 목록
- **데이터 전달 방식:** 파일 기반

#### 4) comparison-judge
- **역할:** 항목 간 동일/차이/불확실 판정
- **입력:** 비교 매트릭스
- **출력:** 항목별 비교 결과
- **데이터 전달 방식:** 파일 기반

#### 5) equivalence-judge
- **역할:** 차이의 실무적 의미를 해석하고 대체 적합성 판단
- **입력:** 비교 결과, 패키지/핀/조건 정보, 보강 정보
- **출력:** 항목별/종합 판단 결과
- **데이터 전달 방식:** 파일 기반

#### 6) report-writer
- **역할:** Excel 및 Word 보고서의 서술 구조와 표시 우선순위 결정
- **입력:** 비교 결과, crop index
- **출력:** 보고서 작성용 구조 데이터, 최종 문장
- **데이터 전달 방식:** 파일 기반

#### 7) web-researcher
- **역할:** 제조사 권장 대체품, 제품 상태, 공식 공지 수집
- **입력:** Part number
- **출력:** 보강 정보 JSON
- **데이터 전달 방식:** 파일 기반

#### 8) qa-reviewer
- **역할:** QA 트리거 발생 시 실제 사용자용 실행 경로로 전체 시스템을 재실행하고, 산출물을 다시 읽어 누락/오류/불일치를 검출하며, 특히 실제 데이터시트 대비 모든 스펙 추출 완전성과 엑셀 셀 내부 이미지 삽입 상태를 검증한 뒤 재코딩 피드백 생성
- **입력:** 전체 산출물, QA 샘플 데이터시트, Plan 문서
- **출력:** validation report, qa_validation_report, rework_feedback
- **데이터 전달 방식:** 파일 기반

## 3.5 스킬/스크립트 파일 목록, 역할, 트리거 조건

### `pdf-datasheet-reader`
- **역할:** PDF에서 텍스트 블록, 표 후보, 페이지 이미지, 좌표 정보를 추출
- **트리거 조건:** 데이터시트 구조 분석 또는 스펙 추출 시작 시

### `evidence-cropper`
- **역할:** 근거 bbox 기반 고해상도 crop 생성
- **트리거 조건:** 각 스펙 항목에 근거 이미지가 필요할 때

### `spec-normalizer`
- **역할:** 스펙명 정규화, 단위 환산, 카테고리 정리
- **트리거 조건:** original/candidate raw spec 추출 완료 후

### `comparison-report-builder`
- **역할:** Excel/Word 문서 생성 및 이미지 배치
- **트리거 조건:** 비교 결과 확정 후

### `manufacturer-web-research`
- **역할:** 제조사/공식 채널 정보 검색 및 정리
- **트리거 조건:** 비교 결과 요약 전 보강 정보가 필요할 때

### `qa-system-runner`
- **역할:** QA 샘플 입력으로 실제 사용자용 전체 시스템을 실행하고 산출물을 다시 읽어 검증 리포트 생성. 실제 데이터시트 대비 스펙 완전성과 엑셀 셀 내부 이미지 삽입 상태도 점검
- **트리거 조건:** 사용자 요청, 코드 수정, 문서 수정, 릴리스 후보 산출물 제출 직전

## 4. 검증 패턴 설계

## 4.1 단계별 검증 유형 적용표

| 단계 | 검증 유형 | 이유 |
|---|---|---|
| 입력 검증 | 규칙 기반 | 파일 존재, 폴더 구조, 파일명 규칙 확인 가능 |
| 섹션 분석 | LLM 자기 검증 + 규칙 기반 | 페이지 누락 여부와 의미 분류 둘 다 필요 |
| 스펙 추출 | 규칙 기반 + LLM 자기 검증 + 사람 검토(선택) | 누락 탐지와 정성 판단이 함께 필요 |
| crop 생성 | 규칙 기반 + LLM 자기 검증 | 각 spec-evidence 연결성과 crop 적절성 확인 필요 |
| 비교 매핑 | 규칙 기반 + LLM 자기 검증 | 모든 spec 포함 여부와 의미 매핑 정확성 필요 |
| 적합성 판단 | LLM 자기 검증 + 사람 검토 | 정성적 위험 판단이 필요 |
| Excel 생성 | 스키마 검증 + 규칙 기반 | 시트, 헤더, 이미지 삽입 확인 가능 |
| Word 생성 | LLM 자기 검증 + 사람 검토 | 문장 품질과 결론 적절성 검토 필요 |
| 최종 검증 | 종합 검증 | 전체 일관성 및 누락 방지 |
| 필수 QA 실행 | 규칙 기반 + 스키마 검증 + LLM 자기 검증 | 실제 사용자용 실행 경로 검증, 실제 데이터시트 대비 모든 스펙 추출 완전성 점검, Excel 셀 내부 이미지 삽입 상태 점검이 모두 필요 |

## 4.2 누락 방지 검증 원칙
본 시스템의 핵심 검증 원칙은 **“모든 페이지를 기준으로 역방향 검증하는 것”**이다.

즉,
1. 추출된 spec 목록을 검토하는 정방향 검증뿐 아니라,
2. 데이터시트 각 페이지를 다시 훑으며 **아직 구조화되지 않은 표/문장/도표/주석이 없는지** 역검토해야 한다.

이 원칙은 특히 다음 영역에 중요하다.
- 표 하단 footnote
- 그래프 축 조건
- 패키지 치수 표의 주석
- 절대 최대 정격과 권장 동작조건의 차이
- 타이밍 파라미터 측정 조건
- 전기적 특성 표의 테스트 조건

## 5.1 실패 처리 패턴 적용

| 상황 | 처리 방식 |

| 공식 웹정보 미발견 | 스킵 + 로그 |
| 비교 매핑 충돌 | 에스컬레이션 |

| QA 산출물에서 누락/오류 발견 | 재코딩 피드백 생성 후 재실행 |
| QA 입력 샘플 부적절 | 에스컬레이션 |

## 5.2 재시도 원칙
- 단순 형식 오류/누락: 최대 2회 자동 재시도
- 의미 불확실성/판단 충돌: 재시도보다 에스컬레이션 우선
- 선택적 보강 정보 실패: 로그 남기고 진행 가능
- QA 실패: 수정 피드백 반영 후 동일 QA 경로로 반드시 재실행

## 5.3 에스컬레이션 조건
다음 조건 중 하나라도 만족하면 사람 검토 대상으로 올린다.

- 스펙이 원문상 존재하는데 구조화에 실패함
- 하나의 스펙이 여러 해석으로 충돌함
- package/pin/function 차이가 실사용 리스크인지 AI가 확신하지 못함
- 제조사 공식 문서 간 정보 충돌 발생
- 최종 대체 적합성 결론이 “가능”과 “비추천” 사이에서 모호함
- QA 샘플이 Plan 문서의 필수 요구사항을 충분히 검증하지 못함

## 6. 설계 판단 및 권장 기술 방향

## 6.1 스크립트 최소화 원칙에 대한 해석
사용자 요구사항은 “스크립트 사용 최소화, 최대한 Claude Code 에이전트가 수행”이다.  
그러나 아래 업무는 결정론적이므로 스크립트가 맡는 것이 바람직하다.

- 파일 탐색, 경로 관리
- PDF에서 bbox 기준 crop 렌더링
- xlsx/docx 파일 생성
- 셀 내 이미지 배치
- JSON 저장 및 로드

반대로 아래 업무는 AI 중심으로 설계해야 한다.

- 데이터시트 섹션 의미 해석
- 스펙 항목의 의미 추출
- 주석/조건의 중요도 판단
- 서로 다른 표현의 동등성 판단
- 차이가 실제 대체 리스크인지 해석
- 종합 의견 작성

즉 본 시스템은 **스크립트 최소화 = 도구 기능 최소 세트만 유지하고, 판단 중심은 에이전트가 담당하는 구조**로 정의한다.

## 6.2 QA 설계 원칙
QA는 테스트 전용 분기 코드가 아니라 **실제 사용자용 전체 시스템의 리허설 실행**이어야 한다.

### 원칙
1. QA는 별도 구현체를 만들지 않는다.
2. QA는 실제 사용자에게 제출할 때 사용하는 동일 코드, 동일 명령, 동일 산출물 경로 규칙을 사용한다.
3. QA 입력은 별도 샘플 폴더에 두되, 실행 로직은 사용자 실행과 동일해야 한다.
4. QA 에이전트는 생성된 Excel/Word를 다시 읽고, Plan 문서를 체크리스트로 삼아 누락을 역검증한다.
5. QA의 최우선 검증 항목은 **실제 데이터시트 대비 모든 스펙 추출 완전성**이다.
6. QA는 근거 이미지가 엑셀에서 단순히 셀 옆에 떠 있는 형태가 아니라, **해당 셀 내부에 삽입/고정된 상태**인지 확인해야 한다.
7. QA 실패는 단순 참고사항이 아니라 재코딩 트리거다.

### 권장 QA 입력 구조
```text
/project-root
 ├── /output
 │   └── /qa
 │       ├── /input
 │       │   ├── /original
 │       │   └── /candidate
 │       ├── /artifacts
 │       └── /reports
```

### QA 결과 해석 원칙
- 통과: 사용자 제출 가능
- 조건부 통과: 경미한 문서/표현 이슈만 존재, 수정 여부 기록
- 실패: 누락/오판단/형식 오류가 있어 재코딩 필요

## 6.3 데이터시트 근거 crop 방식 권장안
근거 crop은 원칙적으로 **PDF 원본 좌표 기반 crop** 방식을 사용한다.

권장 이유:
- 화면 캡처보다 화질 열화가 적음
- 원본 PDF 해상도를 활용 가능
- 텍스트, 표, 그래프, 도면 일부를 정확히 잘라낼 수 있음
- 엑셀 삽입 시 읽기 품질 유지 가능

### 원칙
- 리사이즈보다 원본 보존 우선
- 의미 보존을 위해 필요한 주변 여백은 포함 가능
- 억지 비율 맞춤 금지
- 표라면 머리글/주석 포함 여부를 AI가 판단
- 그래프라면 축, 단위, 조건이 보이도록 crop

## 6.4 비교 판단 원칙
대체 적합성은 단순히 “수치가 비슷한가”로 판단하지 않는다.
반드시 아래 관점을 포함한다.

1. **절대최대정격 일치/상회 여부**
2. **권장 동작 조건의 호환성**
3. **전기 특성의 동일성 또는 허용 가능한 차이 여부**
4. **기능적 호환성**
5. **패키지/핀 호환성**
6. **열특성/타이밍/보호기능 차이**
7. **테스트 조건 차이에 따른 수치 오해 가능성**
8. **공식 제조사 정보의 존재 여부**

### 최종 판정 등급 권장안
- **A: 직접 대체 가능**
- **B: 조건부 대체 가능**
- **C: 주의 필요 / 추가 검증 필요**
- **D: 대체 비추천**

## 7. 향후 확장 포인트
1. 후보 1개가 아닌 다중 후보 비교 지원
2. 회로 용도별 중요 스펙 가중치 적용
3. 특정 부품군(DC/DC, LDO, OpAmp, MCU, MOSFET 등) 전용 비교 템플릿
4. 패키지 도면/핀맵 비교 자동 강화
5. 유통사 재고/가격/리드타임 정보 연계
6. 최종 보고서에 “실제 회로 적용 시 확인할 체크리스트” 자동 생성

## 8. 최종 요약
본 시스템은 Claude Code 기반의 **서브에이전트 오케스트레이션 구조**를 사용하여,

- 기존 부품과 대체품 데이터시트를 **전 페이지 단위로 면밀히 분석**하고,
- **모든 스펙을 100% 누락 없이 추출**하며,
- 각 스펙의 **원본 근거를 고화질 crop**로 연결하고,
- Excel에서 사람이 바로 대조 가능하도록 문서화하고,
- Word 보고서에서 AI가 **실무적인 대체 적합성 판단**을 종합 제공하도록 설계한다.
- 그리고 사용자 요청, 코드 수정, 문서 수정 시에는 **실제 사용자용 코드 경로 그대로 필수 QA를 재수행**하도록 설계한다. 이때 QA는 반드시 실제 데이터시트 대비 모든 스펙 추출 여부와 Excel 셀 내부 이미지 삽입 여부를 핵심 항목으로 검사한다.

핵심 설계 철학은 다음 두 가지다.

1. **누락 없는 스펙 추출**
2. **근거가 즉시 보이는 검토 가능한 판단**

이 두 원칙을 만족하기 위해, 판단은 에이전트가 맡고, crop/문서 생성 등 결정론적 작업만 최소한의 스크립트로 보조한다.
